What Is FreeServicesHub?

FreeServicesHub is a free, open-source service-agent management platform built on .NET 10. It monitors remote Windows services by collecting real-time CPU, memory, and disk metrics, then displays them on a live Blazor dashboard powered by SignalR.

The system is built on the FreeCRM framework — inheriting its authentication, multi-tenancy, plugin system, EF Core data layer, and SignalR infrastructure. Everything new follows the .App. extension-file pattern so the base framework can be updated independently.

Hub

Blazor Server + WebAssembly app. Receives agent heartbeats via SignalR, stores metrics in a multi-provider database (SQL Server, MySQL, PostgreSQL, SQLite, or InMemory), and pushes live updates to connected dashboards.

Agent

.NET Worker Service that runs as a Windows Service. Collects system snapshots every 30 seconds, sends heartbeats to the hub over SignalR, and falls back to HTTP POST + local logging when disconnected.

Installer

Dual-mode CLI tool. Interactive menu for developers; headless mode for CI/CD pipelines. Builds, deploys, configures, and manages the agent as a Windows Service using sc.exe.

Two-Key Trust Chain

Security between the hub and agents uses two cryptographic keys with distinct lifecycles:

Background Services

Three hub-side background services run continuously. Their logs are viewable on the Background Services page in real time.

AgentMonitorService

Polls every 5 seconds. Loads all agents, compares each agent's LastHeartbeat against the stale threshold. Detects status transitions (Online → Stale → Offline) and broadcasts changes to the AgentMonitor SignalR group. Uses exponential backoff on errors (5s × multiplier, max 60s).

DevRegistrationKeySeeder

Runs once on startup in Development mode only. Seeds a known registration key (dev-test-key-seed) into the InMemory database so the Aspire-orchestrated agent can auto-register without manual key generation.

BackgroundProcessor

Configurable scheduled processor. Enabled via BackgroundService:Enabled in appsettings. Supports load balancing by hostname filter. Processes queued work items at a configurable interval.

Deployment Pipeline (12-Step Lifecycle)

STEP  ACTION                        ACTOR           DETAIL
────  ────────────────────────────  ──────────────  ─────────────────────────────────────
  1   Generate registration keys    Pipeline → Hub  N keys for N servers, 24hr, one-use
  2   Shutdown agents               Hub → Agents    "Shutdown" via SignalR
  3   Agents confirm shutdown       Each agent      Graceful stop, confirm
  4   Hub reports all offline       Hub             Pipeline waits
  5   Shutdown hub                  Pipeline        Stop hub process
  6   Deploy hub                    Pipeline        Install new build, start hub
  7   Hub comes online              Hub             Reconnect to DB, hashes persist
  8   Deploy agents                 Pipeline        Install on each server, inject key
  9   Agents start                  Each agent      Read registration key from config
 10   Agents register               Agent → Hub     One-time key consumed, token returned
 11   Agents store token            Each agent      Save to config, begin heartbeat
 12   Dashboard shows online        Hub             AgentMonitorService pushes to UI

Timing: ~5 minutes total. Azure DevOps YAML pipeline with parallel build stages. Web app, agent, and installer build in parallel, then deploy sequentially.

Agent Reconnection Strategy

The agent uses ExponentialBackoffRetryPolicy : IRetryPolicy that never gives up:

AttemptDelayAction
12sReconnect via SignalR
24sReconnect via SignalR
38sReconnect via SignalR
416sReconnect via SignalR
5+30s (cap)Reconnect via SignalR

When disconnected, the agent buffers heartbeats locally and flushes them on reconnect. If SignalR is unavailable, it falls back to HTTP POST. If both fail, it logs to agent.log (rolling file, 5 MB cap) and retries next cycle.
