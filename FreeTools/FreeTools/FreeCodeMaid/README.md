# FreeCodeMaid

A Roslyn-based **C# member reorganizer** for the FreeTools suite. It fills the one gap that
`.editorconfig` + `dotnet format` cannot: **reordering the members of a type** (fields, properties,
methods…) — CodeMaid's signature feature — while **preserving each member's attached comment /
XML-doc block**, following a **modular, customizable rule set**.

It is the deliberate counterpart to the per-language style references in GuidesV2 (`055`–`059`):
those documents describe the convention; FreeCodeMaid enforces the reorderable part of it.

## What it does

For every type in every `.cs` file under the target, it sorts members by:

1. **Kind** — the `KindOrder` table (const → field → constructor → … → method → nested types).
2. *(optional)* **Accessibility** — `GroupByVisibility` (off by default; FreeCRM interleaves).
3. *(optional)* **Static-first** — `StaticMembersFirst` (off by default).
4. **Alphabetical** within the group — with a leading `_` ignored, so `_connectionString` sorts as
   `connectionString` and a non-underscore DI field like `data` slots in by its bare name (exactly
   the FreeCRM field convention).

Each member moves **with the comment block directly above it**. Constructors and operators keep their
original relative order (`DoNotSortKinds`). Consecutive fields stay dense; a blank line is used
between block members.

## Safety (it would rather do nothing than do harm)

- **Skips types containing `#region` / `#if` / `#pragma`** — reordering across those scrambles them.
- **Skips vendored / generated / build output** (`Exclude`) and **by-purpose / generated files**
  (`SkipFiles`) — including FreeCRM's `DataObjects*.cs` (PK-first DTOs) and `DataController.cs`
  (purposeful field block). Imported production code is never touched.
- **Skips machine-generated files** (`<auto-generated>` header).
- **Only rewrites a type when the member order actually changes** — whitespace is left to the formatter.
- **Aborts any file** where a post-rewrite check finds a member went missing or a syntax error was
  introduced. It re-parses its own output and compares the full member set before writing.

## Caveats (read before `--apply`-ing broadly)

- **Field initializer order.** Reordering fields is safe *unless* one field's initializer reads another
  field of the same type (`int b = a + 1;`). That still compiles but can change runtime behavior, and a
  build won't catch it. FreeCRM's fields are independent (`= String.Empty` / `= new()` / `= null`), so it
  is safe there — but review field moves in code you don't know. (A future version can auto-detect
  cross-field references and skip those.)
- **`.razor` files are not processed.** FreeCodeMaid reorders `.cs` only; member order inside a Razor
  `@code` block is left to the author (Razor pages are by-purpose anyway — see GuidesV2 doc 056).
- **Verified, not assumed.** On the real FreeCRM repo this reorganized 31 files; the result **builds with
  0 errors** and is **idempotent** (a second run changes nothing).

## Usage

```
FreeCodeMaid <path> [options]
```

- `<path>` — a `.cs` file, a directory, or a `.sln` / `.slnx` / `.csproj` (defaults to the current dir).
- `--apply`, `--write` — rewrite files in place. Without it, runs as a **dry run**.
- `--check` — dry run (the default): reports what would change and exits `1` if anything would.
- `--config=<file>` — use a specific `freecodemaid.json`. Otherwise the tool searches upward from
  `<path>`, then falls back to built-in defaults.
- `--verbose`, `-v` — also list skipped/unchanged files.
- `--help`, `-h`.

Exit codes: `0` clean/applied · `1` changes needed (check mode — handy as a CI gate) · `2` error.

### Examples

```powershell
# Dry-run the whole FreeCRM repo (nothing is written)
dotnet run --project FreeCodeMaid -- C:\src\FreeCRM

# Apply to one project, writing changes
dotnet run --project FreeCodeMaid -- C:\src\FreeCRM\CRM.DataAccess --apply

# CI gate: fail the build if anything is out of order
dotnet run --project FreeCodeMaid -- C:\src\FreeCRM --check
```

## Configuration

Drop a `freecodemaid.json` in the repo root (a fully-commented copy ships next to this README). Every
rule above is adjustable: the kind order, alphabetical vs. visibility grouping, static-first, blank-line
spacing, and the exclude / skip globs. Defaults are tuned to the FreeCRM convention.
